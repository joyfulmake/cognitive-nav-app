North Star

Traditional education tests memory. This app tests cognitive navigation. The act of formulating the exact right question at the right depth requires a structural, near-unconscious understanding of the system's inflection points. You are building a mirror for the mind, not a quiz engine. The user's question is the output being graded. The answer is just the reward for reaching the right depth.

01

The Depth Rubric

Four layers. Every question a user types is evaluated against these four. The AI does not answer Layer 1 or 2 questions. It pricks. It redirects. It forces reformulation until the user reaches Layer 3 or 4. Vedic philosophy calls this परिप्रश्न (Pariprashna): deep, relentless, investigative inquiry.

01
Factual

What / Who

Surface recall. Asks for a named fact or entity. No structural understanding required. A dictionary could answer this.

Example: "What is TCP?"
02
Relational

How / Why

Cause and effect. Asks how components interact or why something happens. Shows basic system awareness.

Example: "Why does TCP do a 3-way handshake?"
03
Systemic

What If

Edge cases, failure modes, structural tensions. User is navigating the system's weak points. Expert-level cognition.

Example: "What if the SYN-ACK is lost in a lossy network and retransmit timer fires during exponential backoff?"
04
Wisdom

Why This System At All

Asks about the design philosophy. Challenges assumptions. Sees the system from outside itself. This is where mastery lives.

Example: "Why was reliability placed in the transport layer rather than the network layer, and what does that assume about end-host intelligence?"
02

The Prick-Action Loop

When a user submits a shallow question, the AI does not answer it. It identifies the exact variable the user ignored and delivers a targeted prick. No judgment. Just a structural redirect. The user reformulates. Loop repeats until depth is reached.

Step 1

User submits question

Any topic. Any domain. The question is tokenized, semantically embedded, and scored against the depth rubric by the local LLM judge model.

Step 2

Depth classified + prick generated

If Layer 1 or 2: The LLM does not reveal the answer. It names the structural variable the user missed and asks a redirecting question. If Layer 3 or 4: full answer + depth acknowledgment.

Step 3

User reformulates

The loop restarts. Each reformulation is stored. Progress is measured by the trajectory across reformulations, not just the final question. The journey IS the grade.

Full Tech Stack

No defaults. No vendor lock-in. No fragile REST-only architectures. Every choice below is justified against the alternative and selected for resilience, offline capability, and global scale.

Layer Technology Reason Default Rejected
Frontend React Native (Expo) + PWA fallback Single codebase for iOS, Android, and web. Offline-first with service workers on PWA. Expo's managed workflow handles OTA updates without app store friction. Flutter (Dart ecosystem is smaller, fewer AI/ML libs). Native iOS/Android (double codebase cost).
State Mgmt Zustand + TanStack Query Zustand is feather-light (2KB), no boilerplate. TanStack Query handles async state, caching, optimistic updates, and offline queue natively. Redux (verbose, overkill). MobX (magic = hard to debug).
Primary DB CouchDB + PouchDB (client-side sync) CouchDB is a NoSQL document store with built-in multi-master replication. PouchDB is its browser/mobile twin. Together they give you offline-first sync with zero custom code. When the device is offline, PouchDB writes locally. When online, it syncs automatically via CouchDB's conflict resolution protocol. PostgreSQL (relational, no native offline sync). Firebase (vendor lock-in, expensive at scale, data leaves your control). MongoDB Atlas (no offline sync natively).
Cache Layer Redis Cluster (server-side) Session data, leaderboard snapshots, recently-fetched knowledge graphs, and AI evaluation caches. TTL-based auto-expiry. Redis Cluster gives horizontal scaling without single-point failure. Memcached (no persistence, no cluster-native support).
Vector Store Qdrant (self-hosted) Stores semantic embeddings of user questions and knowledge graph nodes. Enables nearest-neighbor search to find similar past questions, track depth trajectory, and suggest depth nudges. Self-hosted means your embeddings never leave your infrastructure. HNSW index is faster than FAISS at production scale with filtering. Pinecone (managed, expensive, data on third-party servers). Weaviate (heavier, more complex ops). FAISS (no native server, just a library).
AI Judge Llama 3.1 70B (quantized, self-hosted via Ollama or vLLM) Runs on your own GPU server. No per-token cost. Fine-tuned on Socratic evaluation datasets and the depth rubric. vLLM gives 20x throughput vs naive inference via PagedAttention. For very constrained offline mode: Llama 3.2 3B on device. OpenAI/Anthropic API (pay-per-token, data leaves device, no offline). GPT-4 (cost at scale is prohibitive for education apps globally).
Embeddings nomic-embed-text (self-hosted via Ollama) 8192 context window. Outperforms OpenAI Ada-002 on retrieval benchmarks. Runs fully locally. Used for question embedding, knowledge graph embedding, and similarity search in Qdrant. OpenAI Embeddings (API cost, data exposure). Cohere Embed (same issues).
Ingestion Engine Python (Scrapy + Playwright-stealth + asyncio) Scrapy handles bulk crawls. Playwright-stealth handles JS-heavy pages and evades bot detection via TLS fingerprint spoofing (curl-impersonate wrapped). Gaussian jitter on request timing to mimic human behavior. Async pipelines so scraping and embedding happen in parallel. BeautifulSoup alone (no JS rendering). Puppeteer (Node, harder to integrate with Python ML pipeline).
Queue / Jobs BullMQ (Redis-backed) Background job queue for: scraping tasks, embedding generation, report generation, and offline sync conflict resolution. Priority queues let real-time user evaluation preempt background scraping. Dead letter queues for failed jobs with retry logic. Celery (heavier, requires RabbitMQ or Redis anyway). AWS SQS (vendor lock-in, cost).
API Layer Fastify (Node.js) + GraphQL (Pothos) Fastify is 3x faster than Express with schema validation built in. GraphQL lets the client declare exactly what data it needs, reducing overfetch. Critical for mobile bandwidth efficiency. Pothos gives type-safe schema definition without code-gen friction. REST only (overfetch/underfetch problems on mobile). Express (slow, no validation). Apollo Server (heavy, opinionated).
Auth Lucia Auth (self-hosted) + Passkey support Lucia is a minimal, framework-agnostic auth library with session management, PKCE OAuth, and passkey (WebAuthn) support. No third-party auth service. Students in regions with poor connectivity benefit from passkey (no OTP SMS needed). Auth0/Clerk (SaaS cost, data on third-party). Firebase Auth (vendor lock-in).
Infra / Deploy Fly.io (edge machines) + Tigris (S3-compatible object storage) Fly.io runs containers at the edge in 30+ regions. Latency matters for AI evaluation responses. Machines can scale to zero between load. Tigris is globally distributed object storage built on Fly's network. No AWS. No GCP. No cost treadmill. AWS/GCP (complex pricing, cold start latency, overkill for early scale). Vercel (no background jobs, no persistent connections).
Monitoring OpenTelemetry + Grafana + Tempo Full observability: traces, metrics, logs. Grafana dashboards for depth distribution across users (see how many users are stuck at Layer 1 vs reaching Layer 4). Tempo for distributed tracing across scrape, embed, evaluate pipeline. DataDog (expensive). Sentry alone (no metrics). No monitoring at all (dangerous at education scale).

Why Documents, Not Rows
A user's question session has variable structure: number of reformulations, optional medical exam context, offline vs online origin. SQL schemas break when this structure evolves. CouchDB documents flex. The _rev field handles sync conflicts automatically when a student uses the app offline on mobile and later syncs to desktop. No custom merge logic needed.
Collection: user_questions
Vector: 768-dim embedding of user's question text
Payload: { user_id, topic, depth_score, reformulation_index, timestamp }
Use: Find users asking structurally similar questions. Cluster them. Surface "you're at the same depth as 847 others who got unstuck by thinking about X."
Collection: knowledge_nodes
Vector: 768-dim embedding of concept description
Payload: { topic, concept, depth_layer, exam_board }
Use: When user submits question, find nearest knowledge nodes at the next depth layer. Generate the prick using structural gap between user's question and Layer N+1 concepts.
Search at Evaluation Time
User submits: "What is unconjugated bilirubin?" (Layer 1). The system embeds the question, finds nearest knowledge nodes at Layer 2 and 3. The gap between the user's vector and the Layer 3 concept cluster IS the prick. The LLM then narrates this gap as a targeted redirect question. Depth grading becomes a vector distance problem, not just an LLM classification problem. Faster, cheaper, more consistent.

Bot Evasion Strategy
TLS Fingerprinting: Default Python requests announces itself via JA3/JA4 hash. Use curl-impersonate to match Chrome's TLS handshake exactly.

Behavioral Jitter: Gaussian distribution on delay (mean 2.5s, std 0.8s). Never constant sleep.

IP Rotation: Only rotate on 429 or CAPTCHA. Not on every request (that pattern is detectable).
Medical DB Strategy
Priority XHR capture over HTML parsing. Most modern medical databases (PubMed, Radiopaedia, Amboss) load content via JSON APIs in the network tab. Capture and replay those requests directly.

Respect all robots.txt. Never store PII. Never hammer a server faster than human browsing speed. Rate: max 1 req/2s per domain.

Why Hybrid (Rule + LLM)
The rule-based pre-classifier handles 60% of questions fast (pure Layer 1 "What is X?" or pure Layer 4 patterns are recognizable with regex and keyword trees). The LLM only runs for ambiguous middle cases. This keeps median latency under 300ms for most users and reserves GPU compute for the genuinely hard evaluations.
05

Offline Architecture

Not a degraded mode. A designed mode. Three tiers of offline capability, each with clear boundaries. Students in low-connectivity regions are not second-class users.

Tier 1: Full Offline

Downloaded Topic Packs

User pre-downloads a topic pack (bundled knowledge graph nodes + pre-embedded vectors stored in IndexedDB/PouchDB). Includes Llama 3.2 3B quantized model (GGUF, ~2GB).

  • Full depth evaluation on device
  • Pricks generated locally
  • Session data stored in PouchDB
  • Syncs to CouchDB on reconnect
Tier 2: Partial Offline

Cached Knowledge + Remote AI

Knowledge graph is cached via service worker. AI evaluation still requires server. Useful when on slow connections (2G/edge): knowledge loads from cache, only the small evaluation payload travels over the network.

  • Knowledge always instant
  • Evaluation waits for server
  • TanStack Query retry queue handles timeouts
  • BullMQ on server for offline job queue
Tier 3: Sync on Reconnect

Optimistic UI + Background Sync

If network drops mid-session, the UI shows optimistic state. Questions are queued in PouchDB. On reconnect, PouchDB syncs to CouchDB, server evaluates queued questions, and results are pushed back.

  • Zero data loss guarantee
  • CouchDB conflict resolution automatic
  • Progress notifications on sync
  • Session continuity preserved
06

App 2: Clinical Crucible

Same engine. Radically different content layer. The medical variant replaces general topic knowledge graphs with curriculum-mapped clinical reasoning trees. It targets MBBS Year 1 through NEET SS, USMLE Step 1-3, and UK PLAB.

Curriculum Mapping

Structured Knowledge Trees

The vector database is pre-loaded with 11 organ system trees. Each node is tagged: year level, exam board, depth layer, prerequisite concepts. A NEET PG user gets different prick redirections than a USMLE Step 1 user for the same clinical scenario, because their expected structural depth differs.

Diagnostic Question Mode

Type Your Investigation

Instead of MCQ, the user types the investigation they would order. "LFT panel" scores Layer 1. "Serum unconjugated bilirubin with peripheral smear for target cells and reticulocyte count" scores Layer 3. The AI evaluates the precision and mechanistic alignment of the chosen investigation, not just the name.

Visual Depth Map

Illuminating Anatomy

No badges. No confetti. The UI shows a layered anatomical or biochemical pathway diagram. As the user's questions deepen, the diagram illuminates from surface to core. The depth of illumination IS the progress indicator. At Layer 4, the root mechanism glows.

Exam Board Calibration

Adaptive Depth Expectations

The LLM judge is calibrated per exam board. NEET SS expects surgical decision-making depth. PLAB expects clinical protocol alignment. USMLE Step 1 expects mechanistic biochemical depth. Same question, different depth scores depending on board context. Stored as a payload filter in Qdrant.

Vedic Foundation
तद्विद्धि प्रणिपातेन परिप्रश्नेन सेवया

Bhagavad Gita 4.34. Knowledge is attained through approach, deep investigative inquiry (परिप्रश्न, Pariprashna), and service. The app does not give knowledge. It creates the conditions for the student to arrive at it through the quality of their own inquiry. The prick IS the guru function.

Cognitive Science Foundation

Active question formulation shifts the brain from the Default Mode Network (passive, associative, recall) to the Task Positive Network (active, generative, structural). This shift is required for neuroplasticity and myelination of expert problem-solving pathways. The act of reformulating a question at higher depth is not a game mechanic. It is the learning event itself. (Source: Journal of Cognitive Neuroscience; studies on active vs passive retrieval and DMN-TPN anticorrelation.)

07

Game Feel / UI Philosophy

Core UI Rule
No gamification fluff. No confetti. No streaks designed to manufacture engagement. The only visual reward is depth illumination: the knowledge map gets brighter, deeper, more detailed as the user's questions improve. The screen literally becomes more beautiful as the user thinks better. That is the only score that matters.
Typography Principle
The user's question text is the largest element on screen. It should feel like it carries weight. Monospace font for the user's typed question (signals: this is code, be precise). Serif font for prick text (signals: this is a teacher speaking). Sans-serif for system chrome. The font itself communicates the cognitive register required.
Depth Color Language (consistent across both apps)
Layer 1: Warm white background. No color accent. The system is neutral, unhurried.
Layer 2: Faint green tint emerges. Movement happening.
Layer 3: Blue-green. The system recognizes real structural thinking.
Layer 4: Deep indigo glow. The map is fully illuminated. The root is visible.

The color is not a reward. It is a reading instrument. It tells the user where they are in the knowledge structure, not how well they are performing.
Trajectory Score (not just final score)
A user who starts at Layer 1 and reaches Layer 4 in 3 reformulations earns a higher trajectory score than one who starts at Layer 2 and stays there. The vector of depth scores across reformulations is stored and becomes the primary mastery signal. Stored in CouchDB session document as trajectory_vector.
Concept Coverage Map
Qdrant tracks which concept nodes a user has reached Layer 3 or 4 on. Stored as a user-concept coverage bitmap in CouchDB. The knowledge map UI renders this coverage as illuminated vs dim nodes. A student can see exactly which parts of their knowledge graph they have depth in and which are still surface-level.
Spaced Repetition Integration
Concepts where the user achieved Layer 3+ are scheduled for re-surfacing using a modified SM-2 algorithm stored in CouchDB. Not to retest recall. To test whether the depth holds over time, or whether the user's questions about the same concept regress to Layer 1 after two weeks. Regression is a signal that the structural understanding was not consolidated. The system resurfaces the concept with a harder prick.
The Architecture in One Paragraph

The user types a question. In under 800ms, a local embedding model converts it to a vector, a nearest-neighbor search finds the structural gap between the user's question and the next depth layer, a hybrid rule-LLM system classifies the depth and generates a targeted prick, and the result is stored in a document database that syncs offline automatically. The knowledge that powers all of this is scraped from the web by a stealth crawler that mimics human TLS fingerprints and browsing rhythms, converted to vector embeddings, and stored in a self-hosted vector database that never sends your data to a third party. The entire system runs on the edge in 30+ regions. Students with no internet access use a 2GB quantized model on device. Nothing requires AWS. Nothing requires a paid API key. This is education infrastructure that can actually reach the world.
